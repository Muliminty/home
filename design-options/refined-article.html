用 GitHub Issues 搭建个人公开时间线

不需要 CMS，不需要数据库。用 Issues 的 label 体系做内容分类，用 Astro 生成静态页面，用 GitHub Pages 免费部署。

2026-05-02 · 阅读 6 分钟

我一直想要一个完全属于自己的内容空间。不是微信公众号那样的封闭平台，也不是 Medium 那种越来越重的写作工具。我希望它轻、快、属于我，并且能长期维护下去。

这个想法断断续续想了很久，直到最近重新审视 GitHub Issues 的时候，突然意识到——它几乎是一个完美的轻量级 CMS。

为什么是 GitHub Issues

GitHub Issues 有几个特性让它天然适合做内容管理：

  • 免费的 Markdown 编辑器，支持图片拖拽上传
  • Label 体系可以充当分类和状态管理
  • REST API稳定、文档完善、有分页
  • 不需要额外服务器，数据跟着仓库走
  • 版本历史天然保留，编辑有迹可循

用 Label 来做内容分类特别巧妙——type:article 标记文章，type:moment 标记动态，status:draft 控制发布状态。不需要数据库 schema，不需要管理后台，一切都在 GitHub 上完成。

技术选型

数据在 GitHub，展示层我选了 Astro。原因很简单：

GitHub Pages 只能托管静态文件，而 Astro 是目前最好的静态站点生成器之一。它默认输出零 JavaScript 的 HTML，首屏极快，SEO 友好。

React 只用在需要交互的地方——标签筛选、图片预览。大部分页面是纯 Astro 组件，构建时就把 HTML 生成好了。

数据流

整个数据流非常简单：

GitHub Issues API
  ↓  fetchIssues.ts (分页拉取)
  ↓  parseIssue.ts (label → type/tag/status)
  ↓  parseArticle.ts / parseMoment.ts
  ↓  Zod schema 校验
  ↓  Astro getStaticPaths
  ↓  静态 HTML

这套链路没有状态管理、没有运行时数据请求、没有缓存失效问题。构建一次，产出就是最终页面。

内容模型

我定义了两种内容形态——文章和动态。它们共享同一个数据源，但在展示层有完全不同的视觉形态。

文章就是传统的长文——有标题、描述、正文、阅读时间。动态更像是随手发的状态——一段文字 + 一组图片，不强制标题。

这两种形态在 /updates 页面混合展示，按时间线排列。读者看到的不再是分门别类的博客列表，而是一个人的近况流。

这个项目还在进行中。如果你对用 GitHub Issues 做 CMS 感兴趣，欢迎关注后续更新。
