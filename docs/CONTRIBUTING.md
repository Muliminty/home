# 文档规范

> 定义项目文档的组织结构、命名规则、内容标准和生命周期，防止文档随迭代失控。

## 文档分层

项目文档分四层，每层有明确的读者和存放位置：

| 层 | 位置 | 读者 | 内容 |
|----|------|------|------|
| **根目录** | `README.md` + `CLAUDE.md` | 所有人 / AI | 项目入口 + AI 上下文（仅这两个） |
| **文档目录** | `docs/*.md` | 开发者/贡献者 | 规范、指南、决策记录、路线图 |
| **自动生成** | `docs/superpowers/` | AI 工具 | 工具产物，不手动编辑 |

### 根目录（仅保留 2 个 .md）

| 文件 | 用途 | 更新频率 |
|------|------|----------|
| `README.md` | 项目介绍、快速开始、技术栈 | 有重大功能变更时 |
| `CLAUDE.md` | AI 助手上下文（技术栈 + 功能状态 + 技能路由） | 每次功能完成后同步 |

**其余所有 `.md` 文档统一放 `docs/` 目录。**

### docs/ 目录结构

```txt
docs/
  CONTRIBUTING.md        ← 文档规范（本文件）
  ROADMAP.md             ← 功能清单和开发状态
  REFACTOR_PLAN.md       ← v1.0 重构决策记录（已归档）
  PUBLISH.md             ← 内容发布指南
  ISSUE_LABEL_SPEC.md    ← 标签体系规范
  decisions/             ← 决策记录（ADR）
    YYYY-MM-DD-title.md
  superpowers/           ← AI 工具自动生成（不手动编辑，超过 90 天自动清理）
    plans/
    specs/
```

### src/ 目录文档规则

`src/` 内不放独立文档。代码说明用行内注释或 JSDoc，需要长篇解释的放 `docs/`。

## 命名规则

- **全大写 + 下划线**：规范/指南 — `README.md`、`ROADMAP.md`、`PUBLISH.md`
- **日期前缀 + 小写**：决策记录 — `2026-05-03-slug-strategy.md`
- **不用中文文件名**：避免跨平台路径问题
- **不用空格**：用连字符 `-` 分隔

## 内容标准

### 语言

- 文档正文用**中文**，代码示例、命令、技术名词保留英文
- `CLAUDE.md` 例外：段落标题用英文（AI 工具识别率更高），内容可中英混合

### 交叉引用

文档间引用统一用**从仓库根目录算起的路径**：

- ✅ `docs/ROADMAP.md`、`docs/PUBLISH.md`
- ❌ `./ROADMAP.md`、`ROADMAP.md`（同目录相对路径）
- ❌ `/docs/ROADMAP.md`（绝对路径）

### ADR 决策记录模板

新建决策记录时，放在 `docs/decisions/`，按以下格式：

```markdown
# 决策标题

> 日期：YYYY-MM-DD，状态：accepted / proposed / deprecated

## 背景

为什么需要这个决策。

## 决策

我们选择了什么。

## 替代方案

考虑过但没选的方案，以及为什么没选。

## 后果

这个决策带来的影响。
```

## 文档生命周期

每个文档有三种状态：

```txt
活跃 (active) → 归档 (archived) → 删除 (deleted)
```

### 活跃

- 内容反映当前项目状态
- 有人负责维护（通常是最后修改者）

### 归档

- 内容已过时但有历史参考价值
- 在文档顶部加标记：`> ⚠️ 已归档：本文档描述的是 [旧版本/旧功能]，仅供历史参考。`
- 保留在原位，不移动（避免破坏外部链接）

### 删除

- 内容完全无用，删除并在 git 历史中保留

## 何时新建文档 vs 更新现有

| 场景 | 做法 |
|------|------|
| 功能完成 | 更新 `docs/ROADMAP.md` 打勾 + `CLAUDE.md` 摘要 |
| 新的技术决策 | 在 `docs/decisions/` 新建 ADR |
| 使用方式变更 | 更新 `docs/PUBLISH.md` 或 `README.md` |
| AI 工具产出 | 自动放入 `docs/superpowers/`，不手动编辑 |
| 临时调研/笔记 | **不要**提交到仓库，用本地文件或 Issue |

## CLAUDE.md 维护规则

`CLAUDE.md` 是 AI 每次对话的起点，必须保持**精简和准确**：

1. **不超过 50 行**：超出就把细节移到 `docs/`
2. **功能完成后立即更新**：从 Planned → Done，不要攒着
3. **不放实现细节**：只放“是什么”，不放“怎么做”
4. **Feature status 可扩展性**：Done 只保留最新版本的功能摘要，历史版本归入 `docs/ROADMAP.md`。当单行超过 200 字符时，改为分行列表

## 过时文档清理清单

> 每次清理后更新此表，已处理的条目打勾或删除。

| 文件 | 问题 | 建议 |
|------|------|------|
| `src/config/README.md` | 描述旧终端 v1 配置，与当前项目无关 | 归档标记或删除 |
| `src/content/blogs/*.md` | 旧系统博客文件，新系统不使用 | 确认是否还需要，否则移到 `archive/` |

## 文档审查节奏

| 触发时机 | 检查内容 | 执行方式 |
|----------|----------|----------|
| 每次功能完成 | `docs/ROADMAP.md` 打勾 + `CLAUDE.md` 同步 | 手动，完成功能时立即执行 |
| 每季度 | 扫描 `docs/` 目录，归档过时文档 | 手动，在 `docs/ROADMAP.md` 底部记录日期 |
| 每个大版本 | 审查 `README.md` 是否反映最新状态 | 手动 |
| `superpowers/` 清理 | 删除超过 90 天的自动生成文件 | 随季度审查一起做 |

> 小贴士：在 `docs/ROADMAP.md` 最后一行保留 `Last doc audit: YYYY-MM-DD`，留痕可查。
